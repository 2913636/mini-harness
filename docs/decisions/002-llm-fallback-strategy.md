# ADR-002: LLM 兜底 + 能力缺口检测

**日期**：2026-07-01
**状态**：已采纳

## 背景

在多 Agent 系统中，用户注册的专家可能不覆盖所有任务需要的领域。需要一个策略处理「缺少对应专家」的情况。

## 决策

**LLM 通用兜底 + 能力缺口告知**。缺专家时不阻塞流程，用通用 LLM 顶替执行，同时告知用户「如果注册了 X 专家，质量会更好」。

## 考虑的方案

### 方案 A：硬失败
缺少专家时直接报错，要求用户注册。

- 优点：简单，强制用户完善专家池
- 缺点：首次用户体验差，什么都没有就报错
- 否决原因：用户可能只是想试一下系统

### 方案 B：LLM 兜底 + 质量提醒
缺少专家时用通用 LLM 执行子任务，同时告知缺口。

- 优点：流程不中断，用户体验好；缺口信息引导用户补充专家
- 缺点：LLM 兜底质量不如专业专家
- **采纳**

### 方案 C：自动生成专家
用 LLM 自动创建一个临时专家。

- 优点：最自动化
- 缺点：自动生成的专家不可靠，行为不可预测
- 否决原因：安全风险，专家行为需要用户掌控

## 实现细节

- `Orchestrator._execute_subtask()` 在专家执行失败或无匹配专家时，调用 `_fallback_llm()`
- `Orchestrator.detect_gaps()` 对比子任务所需领域 vs 已注册专家领域，列出缺口
- 缺口信息作为 `gaps` 字段返回，并在 session 中记录 system 消息提醒

## 影响

- Orchestrator 复杂度增加（双路径：专家执行 + LLM 兜底）
- 用户可见的能力缺口列表，引导后续专家注册
