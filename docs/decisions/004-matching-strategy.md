# ADR-004: 选择字符串匹配作为默认专家匹配策略

## 状态

已采纳（Accepted）

## 背景

Orchestrator 在执行子任务前需要将子任务匹配到最合适的专家。需要选择匹配算法的实现策略。

## 决策

**默认使用基于关键词的字符串匹配策略，同时预留可插拔的 embedding 匹配扩展点。**

## 考虑的方案

### 方案 A：纯字符串匹配（✅ 采纳为默认）

- 零依赖，可解释性强
- 按"领域关键词 → 能力关键词 → 描述词重叠"三层加权打分
- 速度极快（O(n*m)，n=专家数，m=关键词数）
- 对中文分词不敏感，子串匹配更鲁棒

### 方案 B：向量语义匹配（🔌 预留为扩展）

- 需要 embedding 模型（sentence-transformers、OpenAI embeddings 等）
- 语义理解更准确（"内存泄漏"能匹配到"性能分析"专家）
- 引入外部依赖和额外延迟
- 可插拔设计：用户实现 `MatchScorer` 协议后注入 `ExpertRegistry`

### 方案 C：LLM 匹配（❌ 否决）

- 每次匹配都需要 LLM 调用，延迟高
- 成本累积（专家数量多时每次拆解都要匹配 N 次）
- 不可预测（LLM 可能返回不一致的结果）

## 影响

- `ExpertRegistry.match()` 使用字符串匹配作为默认实现
- `ExpertRegistry` 新增 `set_match_scorer(scorer)` 方法，允许注入自定义匹配器
- 用户可实现 `MatchScorer` 协议接入 embedding 模型
- 默认行为对零配置用户友好，同时不阻塞高级用户的语义匹配需求
